# Provenance Guard

Provenance Guard is a backend system designed to classify submitted content as either human-written or AI-generated. It utilizes a multi-signal detection pipeline to provide confidence scores, surface transparent labels to users, and manage appeals for misclassified content.

## Architecture Overview

1. **Submission:** A user submits a piece of their writing. Upon hitting submit, this is caught by the Flask web server dispatcher.
2. **Signal 1 (Structural):** Flask passes the writing to the stylometric heuristic tool. This analyzes purely the style and syntax of the text (like sentence length and vocabulary usage), as human text is much more varied than uniform AI text.
3. **Conditional Logic:** If the stylometric score is less than 0.25 (highly varied), the system fast-tracks the result as human. If it is 0.25 or higher, the system requires a "second opinion."
4. **Signal 2 (Semantic):** For scores >= 0.25, the text is passed to the Groq LLM to check semantic patterns, such as AI tendencies in tonal shifts and meaning.
5. **Final Scoring & Logging:** The scores from both tools are averaged into a single final confidence score. Before answering, Flask puts the score, reasoning, and generated transparency label into the audit log database for a permanent record.
6. **Response & Appeal:** Flask responds to the user with the label and score. If the user appeals, they submit their `content_id` and reasoning, which immediately matches to the audit log and flags the status as "under review" for a human reviewer.

## Detection Signals

* **Signal 1: Stylometric Heuristics (Python Math)**
* **What it measures:** Syntactical properties like the standard deviation of sentence lengths.
* **Why it was chosen:** Human text naturally features chaotic, erratic sentence lengths, whereas AI models tend to produce highly uniform sentence structures.
* **What it misses:** It can only detect patterns in syntax, not actual meaning. An AI might mimic varied human speech tendencies structurally, but lack coherent semantic meaning.


* **Signal 2: Groq LLM (llama-3.3-70b-versatile)**
* **What it measures:** Semantic coherence, tonal shifts, and AI-predictable meanings.
* **Why it was chosen:** It provides a necessary semantic counterweight to the purely mathematical stylometric check.
* **What it misses:** It can miss artistic structural intent. For instance, a poem or song with a repeated section like a chorus might be flagged as AI because the LLM struggles with that specific human artistic structure.



## Confidence Scoring & Uncertainty

To ensure that false positives (falsely accusing a human of using AI) are minimized, the system leverages a "fast-track". A single stylometric check is not enough to accuse a user of AI generation, but it *is* enough to clear them if the text is highly varied (< 0.25).

This logic produces meaningful variance rather than just a binary 0 or 1.

**Validation Examples (Actual Output):**

* **High-Confidence Human Submission:** "ok so i finally tried that new ramen place downtown and honestly? underwhelming. the broth was fine but they put WAY too much sodium in it and i was thirsty for like three hours after."
* **Resulting Score:** `0.18` *(Fast-tracked as human, LLM bypassed)*


* **Lower-Confidence (Uncertain) Submission:** "Artificial intelligence represents a transformative paradigm shift in modern society. It is important to note that while the benefits of AI are numerous, it is equally essential to consider the ethical implications."
* **Resulting Score:** `0.65` *(Stylometric: 0.5 | LLM: 0.8 -> Averaged to 0.65)*



## Transparency Labels

Based on the final confidence score, the reader will see exactly one of the following labels:

| Score Range | Transparency Label Text |
| --- | --- |
| **0.00 – 0.24** | "Analysis indicates this content was highly likely written by a human." |
| **0.25 – 0.74** | "Analysis shows mixed signals. The origin of this content is uncertain." |
| **0.75 – 1.00** | "Analysis indicates this content was highly likely generated by AI." |

## Rate Limiting

**Limits Chosen:** `10 per minute; 100 per day`
**Reasoning:** On a creative writing platform, a human author might legitimately submit a few chapters or poems in a single batch (hence 10 per minute). However, limiting it to 100 per day easily prevents an automated adversary or bot from flooding the system with synthetic content to test the detection boundaries.

## Known Limitations

The system will likely struggle with repetitive creative formatting. A human-written song that includes a heavily repeated section (like a chorus) will cause the stylometric tool to flag the text for "uniform AI syntax" due to the low mathematical variance in sentence length. The LLM might simultaneously miss the artistic structural intention, leading to an unfair "Uncertain" or "AI" flag for a purely human artist.

## Spec Reflection

* **How the spec helped:** Defining the exact mathematical thresholds (the `< 0.25` fast-track) in the spec beforehand made implementing the conditional logic in Python incredibly smooth. There was no ambiguity on when to call the LLM.
* **How the implementation diverged:** Initially, the plan was just to return the score and the label to the user. During implementation, I decided to explicitly return a `reasoning` string in the API payload so the client-side could see exactly which signals were used (e.g., "Fast-tracked human" vs. "Scores averaged"), which greatly aided in debugging and transparency.

## AI Usage

The AI successfully wrote the variance math, but I had to override and revise the endpoint port to `5001` because Apple's AirPlay receiver was blocking port `5000` on my machine after doing some research.
* **Instance 2:** I asked the AI to generate the combination logic and the Groq LLM API call using `llama-3.3-70b-versatile`. The AI provided the correct `if/else` logic based on my spec, but I had to ensure the prompt specifically forced the LLM to return *only* a float between 0.0 and 1.0 to prevent the JSON parser from crashing if the LLM became overly verbose.
